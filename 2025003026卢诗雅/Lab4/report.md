# Lab4：Dice Roller 交互应用与 Android Studio 调试

## 一、应用界面结构说明

本应用是基于 **Jetpack Compose** 开发的简易掷骰子安卓应用，整体界面采用**单页面垂直布局（Column）** 设计，核心界面组件仅包含两个核心元素，布局简洁且居中展示：

1. **骰子图片组件**：位于界面垂直居中位置，默认显示点数为1的骰子图片，用于直观展示掷骰子结果；

2. **掷骰子按钮**：位于骰子图片下方，通过间距（Spacer）与图片分隔，按钮文字为「Roll」，点击后触发掷骰子逻辑；

### 整体布局规则

- 根布局使用 `Column` 垂直排列组件，设置**水平居中对齐**；

- 整体界面填充屏幕（`fillMaxSize`），并将内部内容居中包裹（`wrapContentSize`），保证在不同设备上都居中显示；

- 图片与按钮之间添加 16dp 的间距，优化界面视觉效果。

## 二、Jetpack Compose 状态保存骰子结果的实现

应用通过 **Compose 状态管理机制** 保存和更新骰子点数，核心代码如下：

```Kotlin

// 定义可观察状态变量，默认值为1
var result by remember { mutableStateOf(1) }
```

### 实现原理

1. **`mutableStateOf(1)`**：创建一个**可变的状态对象**，初始值为骰子点数1，这是状态的「数据源」；

2. **`remember`**：将状态对象缓存，避免界面重组时状态被重置，保证状态的持久性；

3. **`by`** ** 委托**：简化状态的读写操作，直接通过 `result` 变量获取/修改状态值；

当按钮点击修改 `result` 时，Compose 会自动检测到状态变化，触发**界面重组**，刷新骰子图片。

## 三、根据点数切换图片资源的逻辑

应用通过 **when 条件表达式** 匹配骰子点数，动态绑定对应的图片资源，核心代码：

```Kotlin

val imageResource = when (result) {
    1 -> R.drawable.dice_1
    2 -> R.drawable.dice_2
    3 -> R.drawable.dice_3
    4 -> R.drawable.dice_4
    5 -> R.drawable.dice_5
    else -> R.drawable.dice_6
}
```

### 执行流程

1. 状态变量 `result` 存储当前骰子点数（1-6）；

2. `when` 表达式根据 `result` 的值，匹配对应的骰子图片资源（`R.drawable.dice_x`）；

3. 将匹配到的图片资源赋值给 `imageResource`，传入 `Image` 组件中渲染；

4. 状态更新时，`when` 会重新执行，自动切换为新点数对应的图片。

## 四、断点设置与观测内容

在 Android Studio 中对核心逻辑设置断点，调试应用运行状态，断点位置及观测目标如下：

|断点位置|观测内容|
|---|---|
|`var result by remember { mutableStateOf(1) }`|观测状态变量的初始化过程，确认初始值为1|
|`val imageResource = when (result) { ... }`|观测 `result` 点数与图片资源的匹配关系，验证资源切换是否正确|
|`Button(onClick = { result = (1..6).random() })`|观测按钮点击事件触发后，`result` 变量的随机赋值结果|
|`Image` 组件渲染行|观测图片资源是否正确加载，`contentDescription` 是否匹配点数|

### 调试结论

- 初始化时 `result` 固定为1，对应加载 `dice_1` 图片；

- 按钮点击后，`result` 被赋值为1-6的随机数，`when` 表达式匹配到对应图片；

- 变量值与界面展示的图片、点数完全一致。

## 五、Step Into、Step Over、Step Out 使用体会

在调试过程中使用了安卓 Studio 调试的三个核心快捷键，总结使用场景和体会：

1. **Step Over（单步跳过）**

    - 用途：逐行执行当前代码，**不进入函数/方法内部**；

    - 场景：执行按钮点击逻辑、状态赋值代码时使用，快速查看当前行执行结果；

    - 体会：最常用的调试操作，适合跟踪主线业务逻辑，效率最高。

2. **Step Into（单步跳入）**

    - 用途：进入当前行调用的**函数/方法内部**执行；

    - 场景：跟踪 `mutableStateOf`、`remember` 等 Compose 内置函数时使用；

    - 体会：适合深入探究底层逻辑，但日常业务调试中使用频率较低。

3. **Step Out（单步跳出）**

    - 用途：跳出当前进入的函数/方法，**返回上一级调用处**；

    - 场景：跳入 Compose 源码后，快速返回业务代码，避免陷入底层细节；

    - 体会：配合 Step Into 使用，快速回归核心业务逻辑，提升调试效率。

## 六、遇到的问题与解决过程

### 问题1：按钮点击后图片不刷新

- **问题描述**：点击「Roll」按钮，骰子图片无变化，界面未更新；

- **原因分析**：未使用 `remember` 缓存状态变量，界面重组时 `result` 被重置为初始值；

- **解决方案**：使用 `var result by remember { mutableStateOf(1) }` 替代普通变量，保证状态持久化。

### 问题2：图片资源匹配错误

- **问题描述**：点数为6时，显示错误的图片；

- **原因分析**：`when` 表达式中 `else` 分支未正确绑定 `dice_6`；

- **解决方案**：修正 `else -> R.drawable.dice_6`，确保所有点数都匹配对应资源。

### 问题3：界面未居中显示

- **问题描述**：骰子和按钮紧贴屏幕左上角，布局异常；

- **原因分析**：未给 `Column` 设置 `fillMaxSize` 和居中对齐属性；

- **解决方案**：为根布局添加 `Modifier.fillMaxSize().wrapContentSize(Alignment.Center)`。

## 七、实验结论

1. **按钮点击后图片自动刷新的原因**

骰子点数 `result` 是 **Compose 可观察状态**，当按钮点击修改 `result` 时，Compose 编译器会检测到状态变化，自动触发**界面重组**，重新执行 `when` 表达式切换图片，最终实现界面自动刷新。

1. **调试变量与界面结果一致性**

调试器中观测到的 `result` 变量值、`imageResource` 图片资源，与界面展示的骰子图片、点数**完全一致**，验证了状态管理和图片切换逻辑的正确性。

1. **核心技术总结**

本应用核心依赖 Jetpack Compose 的**状态管理**和**声明式UI**特性，通过状态驱动界面更新，无需手动操作视图，简化了安卓界面开发流程。
